# 测试反馈与返工闭环

> **配套主文档：** `工作流_TDD_Subagent分工.md`
> **解决的问题：** 测试报错怎么存、怎么切片、怎么以最小 token 喂给返工 subagent，让它精准修复而不读大日志。

---

## 0. 闭环全景（5 步）

```
[1] test-runner 跑测试，原始报告落盘
        │
        ▼
[2] report-slicer 把大报告切成 N 张"failure card"（每张 ≤ 2KB）
        │
        ▼
[3] orchestrator 读 failure card 索引，按文件归并 + 决策派单
        │
        ▼
[4] bug-fixer 收到「failure card + 可疑实现文件」自包含 prompt，改 1 个文件
        │
        ▼
[5] test-runner 仅重跑被修的测试 → 通过则关卡片，失败则 attempt+1 回到 [3]
```

**核心思想**：subagent 永远不读完整日志，只读专属于自己的"失败卡"。

---

## 1. 报告存储目录结构

```
reports/
├── runs/
│   ├── 2026-05-03T14-22-01_run-007/         ← 每次跑测试一个目录（时间戳 + 序号）
│   │   ├── meta.json                         ← 这次跑的元信息
│   │   ├── stdout.log                        ← 原始 stdout/stderr（人类查问题用）
│   │   ├── vitest-unit.json                  ← vitest --reporter=json
│   │   ├── vitest-integration.json
│   │   ├── playwright.json                   ← playwright --reporter=json
│   │   ├── playwright-html/                  ← playwright html 报告（人类看）
│   │   └── playwright-traces/                ← 失败用例的 trace.zip + screenshot
│   │       ├── flow-register__should-fail.zip
│   │       └── flow-register__should-fail.png
│   └── 2026-05-03T15-08-43_run-008/
│
├── failures/                                 ← 切片后的失败卡（subagent 唯一输入源）
│   ├── _index.json                           ← 当前未关闭卡片清单
│   ├── F-UM-01/
│   │   ├── unit__password-strength__min-length.card.json
│   │   ├── integration__user.register__duplicate-mobile.card.json
│   │   └── e2e__flow-register__can-register-with-valid-input.card.json
│   ├── F-UM-02/
│   └── F-IM-04/
│
├── history/                                  ← 每张卡的修复历史（防死循环）
│   ├── F-UM-01__integration__user.register__duplicate-mobile.history.json
│   └── ...
│
└── latest -> runs/2026-05-03T15-08-43_run-008  ← 软链，永远指向最新
```

### 1.1 命名约定（重要！）

**Failure card 文件名格式：**
```
<test_kind>__<test_file_id>__<test_name_kebab>.card.json
```
- `test_kind`：unit / integration / e2e
- `test_file_id`：从测试文件路径推导的稳定 ID（如 `user.register`）
- `test_name_kebab`：测试名转 kebab（去中文用 spec.json 里的 caseId）

**好处**：文件名本身就是去重键，同一测试的下一次失败会**覆盖**而不是堆积，自然形成"当前状态视图"。

## 2. 测试运行命令（让报告落到对的地方）

### 2.1 Vitest 端

```jsonc
// package.json
{
  "scripts": {
    "test:unit": "vitest run --reporter=json --outputFile=reports/runs/$RUN_ID/vitest-unit.json tests/unit",
    "test:integration": "vitest run --reporter=json --outputFile=reports/runs/$RUN_ID/vitest-integration.json tests/integration",
    "test:targeted": "vitest run --reporter=json --outputFile=reports/runs/$RUN_ID/vitest-targeted.json"
  }
}
```

`$RUN_ID` 由 `test-runner` subagent 启动时生成（`date +%Y-%m-%dT%H-%M-%S`_`run-NNN`），并写到 `meta.json`。

### 2.2 Playwright 端

```ts
// playwright.config.ts
export default defineConfig({
  reporter: [
    ['json', { outputFile: `reports/runs/${process.env.RUN_ID}/playwright.json` }],
    ['html', { outputFolder: `reports/runs/${process.env.RUN_ID}/playwright-html`, open: 'never' }],
    ['list'],
  ],
  use: {
    trace: 'retain-on-failure',          // 失败才留 trace
    screenshot: 'only-on-failure',
    video: 'retain-on-failure',
  },
  outputDir: `reports/runs/${process.env.RUN_ID}/playwright-traces`,
});
```

### 2.3 跑测试的统一入口（test-runner subagent 用）

```bash
# 由 test-runner 包装成一条命令
RUN_ID=$(date +%Y-%m-%dT%H-%M-%S)_run-$(printf %03d $NEXT_NO)
mkdir -p reports/runs/$RUN_ID
export RUN_ID

npm run test:unit         > reports/runs/$RUN_ID/stdout.log 2>&1
npm run test:integration >> reports/runs/$RUN_ID/stdout.log 2>&1
npm run test:e2e         >> reports/runs/$RUN_ID/stdout.log 2>&1

ln -sfn runs/$RUN_ID reports/latest
node scripts/slice-failures.mjs reports/latest   # ← 关键：生成 failure cards
```

## 3. Failure Card —— 把大日志切成小卡片

### 3.1 Card schema（统一格式，含 unit / integration / e2e）

```jsonc
{
  "card_id": "integration__user.register__duplicate-mobile",
  "feature": "F-UM-01",                              // 来自 spec.json 反查
  "kind": "integration",                              // unit | integration | e2e
  "run_id": "2026-05-03T15-08-43_run-008",
  "attempt": 2,                                       // 第几次返工（来自 history）

  "test": {
    "file": "tests/integration/user.router.test.ts",
    "line": 87,
    "name": "注册时手机号重复应报 CONFLICT 错误",
    "describe_path": ["user.router", "register"]
  },

  "failure": {
    "message": "expected error code 'CONFLICT' but received 'BAD_REQUEST'",
    "expected": "TRPCError { code: 'CONFLICT' }",
    "actual":   "TRPCError { code: 'BAD_REQUEST', message: 'mobile invalid' }",
    "stack_trim": [                                   // 仅项目内 frame，删 node_modules
      "src/server/routers/user.router.ts:34:11",
      "tests/integration/user.router.test.ts:91:24"
    ],
    "duration_ms": 47
  },

  "context": {
    "suspect_files": [                                // 给 bug-fixer 看的实现文件
      "src/server/routers/user.router.ts"
    ],
    "frozen_files": [                                 // 仅作类型/契约参考，不准改
      "src/contracts/user.contract.ts",
      "prisma/schema.prisma"
    ],
    "test_file_excerpt": {                            // 仅切失败的那一个 it() 块
      "from_line": 84,
      "to_line": 102,
      "content": "  it('注册时手机号重复应报 CONFLICT 错误', async () => { ... })"
    }
  },

  "artifacts": {                                       // 大文件只放路径，不内联
    "trace_zip": null,                                 // e2e 才有
    "screenshot": null,
    "stdout_excerpt_lines": [1230, 1278]               // stdout.log 里这段的行号
  },

  "history_ref": "history/F-UM-01__integration__user.register__duplicate-mobile.history.json"
}
```

### 3.2 切片脚本职责（`scripts/slice-failures.mjs`）

```
1. 读 vitest-unit.json / vitest-integration.json / playwright.json
2. 遍历每条 failed test
3. 反查：
   - feature 编号（用 spec.json 的 caseId → feature 映射）
   - suspect_files（按"测试文件名 → 实现文件"约定映射；E2E 用 page-map）
   - test_file_excerpt（grep `it('<name>'` 的起止行）
4. 写 reports/failures/<feature>/<card_id>.card.json
5. 更新 reports/failures/_index.json（含未关闭卡片清单）
6. 追加 reports/history/<card_id>.history.json
```

**这个脚本由 bootstrap-agent 一次性写好**，是工程的一部分，不是临时产物。后续所有 subagent 都依赖它。

### 3.3 Card 体积控制

| 字段 | 上限 |
|------|------|
| `failure.message` + `expected` + `actual` | 1KB |
| `stack_trim` | 最多 8 行 |
| `test_file_excerpt.content` | 最多 50 行 |
| `stdout_excerpt_lines` | 仅行号，不内联文本 |

**单卡 ≤ 2KB**，外加可疑实现文件（≤ 6KB），总输入 ≤ 8KB——刚好卡 bug-fixer 预算。

## 4. Orchestrator 决策表（看到 failure 怎么派单）

```
读 reports/failures/_index.json
  │
  ▼
按 card 类型分流：
  │
  ├── kind=unit + attempt ≤ 3              → bug-fixer-light    （改 1 个 service/repo 文件）
  │
  ├── kind=integration + attempt ≤ 3       → bug-fixer-light    （改 1 个 router 文件）
  │
  ├── kind=integration + attempt 4~6       → bug-fixer-deep     （可读 router + service + repo）
  │
  ├── kind=e2e + 选择器问题                 → e2e-test-fixer     （改 spec，不改 src）
  │       判定：actual 是 "selector not found / timeout"
  │
  ├── kind=e2e + UI 问题                   → ui-impl-fixer      （改 1 个 page/component）
  │       判定：actual 是 "expected text X but got Y" 等业务断言
  │
  ├── kind=e2e + flaky                    → flake-detective    （读 trace.zip）
  │       判定：上一轮通过、这一轮失败 → history 里查
  │
  ├── 编译错误（tsc 报错，非 assertion）    → 拒收，回退到 writer 重写
  │       特征：actual 含 "TS2xxx" / "Cannot find name"
  │
  └── attempt > 6                          → 标 BLOCKED，进交付报告"风险"段
```

**关键规则**：
- **同一文件多张卡**先合并：如果 `user.router.ts` 同时挂了 5 张卡，**不派 5 个 fixer**，派 1 个 fixer 一次解决（prompt 里塞 5 张 card）。这避免修一个改坏另一个的级联。
- **跨文件的卡分别派**，可并行（不同 fixer 改不同文件无冲突）。

## 5. Bug-Fixer 的完整 Prompt 模板（关键！）

### 5.1 `bug-fixer-light` 模板（替换花括号即用）

````markdown
# Role: Bug Fixer (Light)

## Your Task
There is/are {N} failing test(s) all pointing to ONE implementation file.
Make the test(s) pass by editing ONLY that file.
You may NOT touch any other file.

## The Single File You May Edit
Path: `{suspect_file_path}`
Current content:
```typescript
{paste full file content here, ≤ 6KB}
```

## Frozen Files (READ for type/signature reference, DO NOT modify)
- `src/contracts/{contract_path}` ← 类型契约
- `prisma/schema.prisma` ← 数据模型
- (only the relevant excerpt is pasted below if < 1KB)

```typescript
{paste contract excerpt}
```

## Failing Test Card(s)

### Card 1 of {N}
- **Test:** `{test.name}` (line {test.line} of `{test.file}`)
- **Expected:** {failure.expected}
- **Actual:** {failure.actual}
- **Stack (project only):**
  - {stack_trim[0]}
  - {stack_trim[1]}
- **Test code excerpt:**
```typescript
{test_file_excerpt.content}
```

### Card 2 of {N}
... (同上)

## Hard Constraints
1. Edit ONLY `{suspect_file_path}`. Any other edit = task FAILED.
2. Do NOT modify any test file.
3. Do NOT modify contract / schema / config.
4. Do NOT add `@ts-ignore`, `.skip`, `.only`, `xit`, `xtest`.
5. Do NOT add new dependencies — output `MISSING_DEP: <name>` if you need one.
6. Keep the file ≤ 300 lines.
7. Do NOT add comments unless explaining a non-obvious invariant (≤ 1 line).

## Verification (you MUST run this and only succeed when it passes)
```bash
RUN_ID=fix-$(date +%s) npx vitest run {test.file} --reporter=json
```
Then run the full integration suite to ensure no regression:
```bash
RUN_ID=fix-$(date +%s) npm run test:integration
```

## Done When
- Both verifications pass.
- Output a 3-line summary: what was wrong, what you changed, line count of diff.

## If You Cannot Fix
Output exactly: `ESCALATE: <one-line root cause that's outside this file>`
The orchestrator will route to bug-fixer-deep with a wider context.
````

### 5.2 `bug-fixer-deep` 比 light 多了什么

只多两点：
1. **可改文件清单从 1 个扩到 ≤ 3 个**（必须列出，不能动其他文件）
2. **附加同 feature 的"通过测试"清单**，提醒不要破坏已绿测试

其他完全相同。

### 5.3 `e2e-test-fixer`（修测试，不修实现）

跟 `bug-fixer-light` 几乎一样，区别：
- 可改文件 = 测试文件本身
- 多附 `tests/fixtures/page-map.md`（页面 selector 字典）
- Hard constraint：**禁止改 `src/`**（防止把"测试错"伪装成"实现错"）

## 6. Playwright 特殊处理

### 6.1 Trace 与 screenshot 的传递方式

**绝不**把 trace.zip / 截图二进制塞进 prompt。做法：

```
e2e card 里只放路径：
  "trace_zip": "reports/runs/2026-05-03T15-08-43_run-008/playwright-traces/flow-register__can-register.zip"
  "screenshot": "...same dir.../flow-register__can-register.png"
```

- **bug-fixer 默认不打开**（节省 token）
- **flake-detective 才会打开 trace**：用 `npx playwright show-trace <path>` 在脚本环境里解析为文本时间线，截取关键 5～10 步给 LLM
- **截图**：仅当 actual 是"视觉/布局类问题"时由编排器作为 image 块附加到 prompt（多模态时才看图）

### 6.2 持久化 state 失效的识别

如果 e2e 突然全挂在登录后第一步：八成是 `.test-state/<role>` 过期/损坏。

`test-runner` 检测规则：
- 失败 ≥ 50% 且失败点都在 "登录后第一个断言" → 自动删 `.test-state/`、跑 `tests/fixtures/setup-roles.ts` 重建
- 重建后再跑一遍；仍失败才生成 failure card

这避免了把"基础设施问题"当成"业务 bug"派给 bug-fixer 浪费 token。

## 7. History 文件防死循环

每张卡片都有一份 history：

```jsonc
// reports/history/F-UM-01__integration__user.register__duplicate-mobile.history.json
{
  "card_id": "integration__user.register__duplicate-mobile",
  "attempts": [
    {
      "attempt": 1,
      "timestamp": "2026-05-03T15:10:22Z",
      "fixer": "bug-fixer-light",
      "edited_file": "src/server/routers/user.router.ts",
      "edit_summary": "把 BAD_REQUEST 改成 CONFLICT",
      "outcome": "STILL_RED",
      "next_failure_diff": "actual still BAD_REQUEST — 怀疑改错了分支"
    },
    {
      "attempt": 2,
      "fixer": "bug-fixer-light",
      "edited_file": "src/server/routers/user.router.ts",
      "edit_summary": "在 catch (e) 里改成检测 Prisma P2002 错误码",
      "outcome": "GREEN",
      "regression": "none"
    }
  ],
  "closed_at": "2026-05-03T15:14:01Z",
  "final_status": "FIXED"
}
```

**派单时**，orchestrator 把过去 attempts 的 `edit_summary` 和 `outcome` 摘要塞进新 fixer 的 prompt（"前面试过 X 但还红，原因是 Y，请换思路"），避免重复尝试同一个无效改法。

## 8. 升级阶梯（Escalation Ladder）

| Attempt | Fixer | 可改文件 | Token 预算 | 触发条件 |
|---------|-------|----------|-----------|----------|
| 1～3 | bug-fixer-light | 1 个 | 8KB | 默认入口 |
| 4～6 | bug-fixer-deep | ≤ 3 个 | 15KB | light 输出 ESCALATE 或 3 次 STILL_RED |
| 7 | flake-detective（仅 e2e） | 测试或 src 任 1 文件 | 20KB（含 trace 摘要） | E2E 间歇性失败 |
| 8+ | 写入交付报告 BLOCKED 段 | — | — | 超出仍未修复 |

BLOCKED 必须人工介入，但**人也不读代码**——人类的动作只能是：
- 重新审视 spec.json，确认测试用例本身是否合理
- 修改 spec.json，触发该 feature 全链路重跑（test-writer → impl）

## 9. 编译错误的特殊路径（重要！）

编译错误（`tsc --noEmit` 报错）**不进 failure card 流程**。它表明实现违反了契约——这是 writer 的问题，不是 bug。

```
test-runner 跑测前先跑 tsc：
  ├── tsc 通过 → 正常跑 vitest/playwright
  └── tsc 失败：
       提取错误 → 定位到出错文件 →
       ├── 出错在 src/contracts/* → 拒收，contract-architect 返工
       ├── 出错在 src/server/routers/* → 拒收，对应 router-impl 返工（不是 bug-fixer）
       ├── 出错在 src/app/* → 拒收，对应 ui-impl 返工
       └── 出错在 tests/* → 拒收，对应 test-writer 返工
```

**为什么不让 bug-fixer 修编译错？** 因为编译错往往意味着对契约的误解，让原 writer 重写比"打补丁"更干净。

## 10. 一个端到端的具体例子

**场景**：`router-impl-user` 写完了 `user.router.ts`，跑测试，1 条 integration 测试失败。

**Step 1** — `test-runner` 执行：
```bash
RUN_ID=2026-05-03T15-08-43_run-008
npm run test:integration > reports/runs/$RUN_ID/stdout.log
```
得到 `vitest-integration.json`，含 1 条 failed。

**Step 2** — `slice-failures.mjs` 自动跑，生成：
```
reports/failures/F-UM-01/integration__user.register__duplicate-mobile.card.json  (1.4 KB)
reports/failures/_index.json  ← 含 1 条未关闭卡
reports/history/F-UM-01__integration__user.register__duplicate-mobile.history.json (空)
```

**Step 3** — orchestrator 读 `_index.json`，决策：
- kind=integration, attempt=1 → bug-fixer-light
- suspect_file = `src/server/routers/user.router.ts`
- 同文件无其他卡 → 单独派 1 个

**Step 4** — orchestrator 用模板 5.1 拼好 prompt（约 7KB），调用：
```
Agent(
  subagent_type="general-purpose",
  description="Fix user.register CONFLICT bug",
  prompt="<填充后的 bug-fixer-light prompt>"
)
```

**Step 5** — bug-fixer 改完 `user.router.ts`，跑：
```bash
RUN_ID=fix-1714748000 npx vitest run tests/integration/user.router.test.ts --reporter=json
```
通过。再跑全量集成测试，无回归。返回："Changed catch block to detect Prisma P2002. Diff +4/-2 lines."

**Step 6** — orchestrator 读返回，更新 history：
```jsonc
{ "attempt": 1, "outcome": "GREEN", "edit_summary": "..." }
```
关闭 card（从 `_index.json` 移除，移动到 `failures/_closed/` 归档）。

**Step 7** — 全部 card 关闭后，进入下一阶段或交付门。

---

## 12. 一句话总结

> **报错存在 `reports/failures/<feature>/<card_id>.card.json`**（每个失败一张 ≤ 2KB 的 JSON 卡）；
> **返工时 orchestrator 把"卡 + 可疑实现文件"塞进 bug-fixer prompt**（自包含 ≤ 8KB）；
> **history 防死循环**，**升级阶梯防卡死**，**编译错走单独路径**直接退回原 writer。
